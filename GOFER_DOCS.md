# Документация проекта `lang-hub` для интеграции с `gofer`

Данный репозиторий (`lang-hub`) представляет собой централизованное хранилище конфигураций, парсеров и утилит (линтеров, форматеров, LSP) для обеспечения работы системы `gofer` с исходным кодом на различных языках программирования.

## Сущности проекта

Проект строится вокруг двух основных сущностей:
1. **Языки (Languages)** — конфигурации для конкретных языков программирования (находятся в директории `list/`).
2. **Инструменты (Tools)** — конфигурации внешних утилит, таких как линтеры, форматеры и LSP-серверы (находятся в директории `tools/`).

Все конфигурации описываются в файлах `manifest.toml`.

---

### 1. Языки (Languages)

Директория `list/` содержит поддиректории для каждого поддерживаемого языка (например, `cpp`, `python`, `rust`). Внутри каждой лежит файл `manifest.toml`.

**Схема и параметры `manifest.toml` для языков:**

* **`[language]`** — Базовая информация о языке:
  * `name` (string) — Название языка.
  * `extensions` (list of strings) — Список расширений файлов, относящихся к языку (например, `["cpp", "cc", "hpp"]`).
  * `aliases` (list of strings) — Альтернативные названия языка.
  * `root_markers` (list of strings) — Файлы или директории, указывающие на корень проекта (например, `["CMakeLists.txt", ".git"]`).

* **`[language.comments]`** — Описание синтаксиса комментариев:
  * `line` (string) — Однострочный комментарий (например, `"//"`).
  * `block` (list of two strings) — Начало и конец многострочного комментария (например, `["/*", "*/"]`).
  * `doc` (string) — Док-комментарий.

* **`[parser]`** — Настройки парсера (обычно Tree-sitter):
  * `type` (string) — Тип парсера (например, `"wasm"`).
  * `source` (string) — Источник (например, `"build"`).
  * `repo` (string) — Имя репозитория Tree-sitter (например, `"tree-sitter/tree-sitter-cpp"`).
  * `download_url` (string) — Прямая ссылка для скачивания скомпилированного парсера (`.wasm` файла).

* **`[lsp]`** — Настройки Language Server (LSP):
  * `name` (string) — Название LSP сервера.
  * `download_url` (string) — Ссылка для скачивания бинарного файла сервера (поддерживает шаблоны `{os}` и `{arch}`).
  * `command` (string) — Команда для запуска сервера.
  * `tool` (string) — Ссылка на инструмент из директории `tools/` (если LSP определён как универсальный инструмент).
  * `init_options` (table) — Специфичные опции инициализации сервера.

* **`[sandbox]`** — Инструкции для песочницы/исполнения:
  * `compile_cmd` (string) — Команда для компиляции (например, `"g++ -std=c++17 -o {out} {src}"`).
  * `run_cmd` (string) — Команда для запуска.
  * `build` (list) — Список аргументов/команд для сборки проекта.
  * `test` (list) — Список аргументов/команд для тестирования проекта.

* **`[indexer]`** — Настройки индексатора `gofer`:
  * `ignore_folders` (list of strings) — Glob-паттерны директорий, которые нужно исключить из индексации (например, `["**/build/**", "**/node_modules/**"]`).

* **`[formatter]`** и **`[linter]`** — Привязка форматеров и линтеров:
  * `tool` (string) — ID инструмента из директории `tools/` (например, `"clang-tidy"` или `"ruff"`).
  * `args` (list of strings) — Аргументы, которые необходимо передать инструменту при вызове (поддерживают шаблоны, такие как `"{src}"`).

---

### 2. Инструменты (Tools)

Директория `tools/` содержит конфигурации независимых утилит, которые могут использоваться различными языками (например, `ruff`, `prettier`, `clangd`). Описываются в своих `manifest.toml`.

**Схема и параметры `manifest.toml` для инструментов:**

* **`[tool]`** — Базовая информация:
  * `name` (string) — Имя инструмента (соответствует названию папки).
  * `description` (string) — Краткое описание утилиты.
  * `capabilities` (list of strings) — Возможности утилиты. Может включать `"formatter"`, `"linter"`, `"lsp"`.

* **`[tool.install.binary]`** — Информация об установке:
  * `url` (string) — Ссылка на скачивание архива/бинарника (часто из релизов GitHub).
  * `executable_name` (string) — Имя исполняемого файла после распаковки.

* Блоки исполнения в зависимости от `capabilities` (**`[formatter]`**, **`[linter]`**, **`[lsp]`**):
  * `command` (string) — Команда для запуска (часто совпадает с `executable_name`).
  * `args` (list of strings) — Стандартные аргументы для запуска инструмента в выбранном режиме (например, `["check", "--fix"]` для линтера или `["format"]` для форматера).

## CI/CD и Автоматизация

Проект использует **GitHub Actions** для поддержки актуальности данных и валидации. Конфигурации находятся в `.github/workflows/`, а вспомогательные скрипты — в `scripts/`.

1. **Валидация манифестов (`validate.yml`)**
   - Запускается при каждом PUSH и Pull Request в ветку `main`.
   - Запускает Python скрипт `scripts/validate_manifests.py`.
   - **Цель:** Убедиться, что структуры всех TOML файлов (в `list/` и `tools/`) корректны и соответствуют ожидаемой схеме, ключи не пропущены, а типы данных верны.

2. **Автоматическое обновление инструментов (`update-tools.yml`)**
   - Запускается раз в неделю (по расписанию: каждое воскресенье) или вручную.
   - Использует скрипт `scripts/auto_updater.py`.
   - **Цель:** Скрипт парсит `tools/*/manifest.toml`, определяет актуальные ссылки `url` из релизов на GitHub, обновляет версии бинарных файлов до последних версий.
   - После обновления автоматически формируется Pull Request с изменениями (например, с названием "chore: Auto-update tool versions").

3. **Сборка парсеров WASM (`build-wasm.yml`)**
   - **Цель:** Автоматизированная загрузка и компиляция исходных кодов с репозиториев языков (Tree-sitter) в бинарный формат WebAssembly (`.wasm`), который впоследствии может быть скачан и использован `gofer` напрямую. Использует скрипт сборки `scripts/build-parser.sh`.

## Рекомендации по интеграции с `gofer`

* `gofer` должен парсить директорию `list/` для получения всей метаинформации о поддерживаемых языках, их расширениях и конфигурациях для запуска и интеграции.
* При вызове команд для линтинга или форматирования `gofer` должен извлечь имя `tool` из манифеста языка, затем найти соответствующий манифест в папке `tools/`, получить ссылку на скачивание (если он ещё не установлен) и объединить параметры запуска (`args` из языкового манифеста + `args` из манифеста утилиты).
* Опция `ignore_folders` в манифесте языков критически важна для оптимизации работы компонента `indexer` (позволяет избежать обхода больших сгенерированных директорий типа `build` или `node_modules`).
